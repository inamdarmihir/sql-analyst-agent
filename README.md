# SQL Analyst Agent

An interpreter-first [Deep Agents](https://github.com/langchain-ai/deepagents) implementation for SQL analytics on a seeded SQLite e-commerce dataset. The agent writes and executes SQL from inside a sandboxed QuickJS interpreter, keeps every intermediate result in interpreter-scoped variables, and only returns compact, reasoned summaries to the LLM's context window — raw rows never leak into the conversation. Narrative analysis of specific result slices is delegated programmatically (via `tools.task`) to sub-agents, and the root agent synthesizes everything into a final written report.

## Why interpreter-first?

Traditional tool-calling agents pass every query result back through the LLM's context: this burns tokens, risks hallucinated row values, and caps the size of data an agent can reasonably analyze. This project instead grants the model a code interpreter with **Programmatic Tool Calling (PTC)** — the model writes TypeScript that calls `tools.executeQuery`, `tools.getSchema`, and `tools.task` directly, storing results as interpreter variables. Only the final, distilled analysis (not raw rows) is returned to the model. This keeps context usage low and lets the agent reason over large result sets safely.

**Request flow:**

1. A user question enters the LangGraph app (`langgraph dev`), routed to the root Deep Agent defined in `agent.py`.
2. The root agent's system prompt instructs it to write interpreter code rather than reasoning directly over raw data.
3. Inside the QuickJS sandbox (`CodeInterpreterMiddleware`), the model's generated code calls `get_schema` once to discover tables/columns, then issues one or more `execute_query` calls — each SELECT-only, comment-stripped, and capped at 500 rows.
4. Query results are held as interpreter variables; only derived facts (aggregates, ratios, short text) are returned to the model's context.
5. Where deeper narrative reasoning is needed on a result slice, the interpreter code calls `tools.task`, which delegates to a `general-purpose` sub-agent — again, invoked from code, not the root model.
6. The root agent writes the final synthesized report to the virtual filesystem via `write_file`, then replies with the report path plus a "Query and Delegation Trace" describing what was queried, analyzed, and delegated.

### Key components

| Component | File | Responsibility |
|---|---|---|
| Agent definition & system prompt | `agent.py` | Wires the model, tools, checkpointer, and `CodeInterpreterMiddleware`; defines operating rules for the agent. |
| SQL tools | `tools/sql_tools.py` | `execute_query` (read-only, SELECT-only, 500-row cap, SQL comment stripping) and `get_schema` (table/column/row-count introspection). |
| Tool exports | `tools/__init__.py` | Re-exports the SQL tools for import by `agent.py`. |
| Dataset seeding | `seed_data.py` | Deterministically generates a synthetic e-commerce SQLite database (customers, products, orders, order_items) using `Faker` with a fixed seed. |
| LangGraph entrypoint | `langgraph.json` | Declares the `agent` graph served by `langgraph dev`. |
| Environment template | `.env.example` | Placeholder for `ANTHROPIC_API_KEY`. |

## Requirements

- Python 3.11+
- [`uv`](https://docs.astral.sh/uv/) for dependency management and running scripts
- An Anthropic API key (or credentials for whichever model provider you configure)

## Setup

```bash
# 1. Install dependencies
uv sync

# 2. Configure environment
cp .env.example .env
# then edit .env and set ANTHROPIC_API_KEY=<your-key>

# 3. Generate the seeded SQLite dataset
uv run python seed_data.py
```

`seed_data.py` creates `data/ecommerce.db` with four tables — `customers`, `products`, `orders`, and `order_items` — populated deterministically (`SEED = 42`) via `Faker`, so re-running it always produces the same dataset.

## Running the agent

```bash
langgraph dev
```

This starts the LangGraph development server using the graph declared in `langgraph.json` (`agent.py:agent`). Open the LangGraph Studio URL printed in the terminal to interact with the agent, or connect via the LangGraph SDK/API.

## Dataset

| Table | Description | Approx. rows |
|---|---|---|
| `customers` | id, name, email, region, signup_date, tier | 2,000 |
| `products` | id, name, category, price, cost, stock_qty | 200 |
| `orders` | id, customer_id, order_date, status, shipping_region | 15,000 |
| `order_items` | id, order_id, product_id, quantity, unit_price, discount_pct | 40,000 |

Regions: `North`, `South`, `East`, `West`. Tiers: `bronze`, `silver`, `gold`, `platinum`. Order statuses: `completed`, `returned`, `cancelled`, `pending`. Order history spans the trailing two years from generation time.

## Example questions

1. Aggregation: "Which region had the highest revenue last quarter?"
2. Cohort analysis: "How does order value change over a customer's first 6 months?"
3. Anomaly: "Which products have unusually high return rates compared to their category average?"
4. Multi-hop: "Are gold-tier customers from the South ordering more in the evenings?"
5. Trend: "Show me month-over-month revenue growth for the last 12 months broken down by tier."

## Swapping the model provider

In `agent.py`, change the `model=` string passed to `create_deep_agent(...)`. For example, switch from Anthropic to another provider/model string supported by your LangChain + Deep Agents setup, and update the corresponding LangChain integration package in `pyproject.toml`.

## Adding a new tool and exposing it via PTC

1. Add a new `@tool` function (for example in `tools/sql_tools.py` or another module).
2. Import and include it in `tools=[...]` inside `create_deep_agent(...)` in `agent.py`.
3. Add the tool name to `CodeInterpreterMiddleware(ptc=[...])` so interpreter code can invoke it programmatically.
4. Update the system prompt guidance with when and how the interpreter should use that tool.

## Troubleshooting

- **`ANTHROPIC_API_KEY` not set**: ensure `.env` exists (copied from `.env.example`) and contains a valid key before running `langgraph dev`.
- **`data/ecommerce.db` missing or stale**: re-run `uv run python seed_data.py`; it drops and recreates all tables.
- **Query rejected with "Only SELECT statements are allowed"**: `execute_query` only permits read-only `SELECT` statements as a safety guard; rewrite the query accordingly.
- **Large result sets truncated**: `execute_query` caps results at 500 rows (`truncated: true` in the response) to keep interpreter memory and downstream analysis bounded — aggregate or filter in SQL instead of relying on full row dumps.
